## 1. 自律性（40%）への評価と提案

### 強み
- 一度起動したら最後まで進む設計（Stage 0〜6）
- `rated_samples.json` からの中断再開
- CLIPキーワード未機能時のフォールバック
- 撮影意図をAIからユーザーへ逆質問し、文脈を自ら補完

### 改善提案

**【Gemini】doctor.sh（自己診断スクリプト）の導入**
`./doctor.sh` で環境（Python / venv / ExifTool / CLIPモデル）を一発チェックし、不足があれば修正コマンドを提案する。「エラーが起きても自分で立て直す」という審査基準に直接対応できる。

**【Gemini】ExifToolをsetup.shに追記**
Stage 6でExifToolを使用しているが、`setup.sh` にインストール手順が含まれていない。`brew install exiftool` の追加、またはエージェントがユーザーへ案内するフローをSKILL.mdに記載する。

**【Gemini】CLIPモデルのプリロード**
Stage 5の初回実行時に340MBのダウンロードが発生する。`setup.sh` 内でダミー実行を行い、キャッシュを事前に作成しておくと、デモ時の自律性が際立つ。

## 2. クオリティ（35%）への評価と提案

### 強み
- ローカル処理（OpenCV/CLIP）とLLM（Claude）のハイブリッドアーキテクチャ
- APIコスト99.99%削減
- 数値的な実績（338分→6分、★4 recall 82%）

### 改善提案

**【Codex】EVALUATION.mdの新規追加（最重要）**
`338分→6分` や `★4 recall 82%` の数値は強力だが、現状では審査員が根拠を即確認しづらい。以下の内容を1ページで明示した `EVALUATION.md` を追加するだけでクオリティ評価が上がる。

| 記載項目 |
|---------|
| データセット条件（撮影枚数・機材） |
| 母数 |
| 正解の定義（例: クライアント納品カット★4の146枚） |
| 旧設計との比較条件 |
| 測定日・バージョン |

**【Codex】テスト不在が弱点**
テスト/spec系ファイルが存在しない。本格的なテストでなくても、smoke testと再現手順をREADMEかEVALUATION.mdに記載するだけで印象が変わる。

**【Gemini】画像-画像モードの技術的強みを前面に**
`score.py` の画像-画像モード（`--mode image`）は、テキストキーワード検索よりも精度が高くAIネイティブなアプローチ。READMEとデモ動画で「**言葉にできない感性を、30枚の画像から直接抽出している**」という点を強調する。

## 3. インパクト（25%）への評価と提案

### 強み
- 「338分→6分」という具体的な数値
- Lightroom連携（XMPメタデータ）で明日から実際に使える
- 家族との時間を取り戻すというストーリー

### 改善提案

**【両者共通】プラグインREADMEのメッセージを統一**
ルートREADMEは「30枚から2500枚を自律選別するエージェント」を前面に出しているが、プラグインREADMEは「Stage 2ダッシュボードが主役」と読める。ハッカソン提出では **「写真家の審美眼を継承して最後まで走る自律エージェント」** に寄せ切ること。

**【Gemini】Lightroom連携の「最後の一押し」**
XMPを書き出すだけでなく、Lightroomで「メタデータを読み込む」ボタンを押した瞬間に星がパッと付く様子をデモ動画に入れる。

## 4. デモ動画について（両者共通）

「精度の高さ」より「**自律復旧**」を1回見せることが最重要。推奨の流れ：

```
依頼 → 必要情報を1回で渡す → Stage1-2自動処理
→ Stage3で6分の人間評価（ここだけ介入）
→ 意図的に中断 → 再開して続きから動く
→ Lightroom に星が入る
```

**デモ時の注意:** インストール場面は映さず、事前セットアップ済みの環境で実施。READMEに「デモは事前セットアップ済み環境で収録」と明記する。
